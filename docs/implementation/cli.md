# CLI and project discovery

The CLI is wiring and command egress, not semantics. Commands resolve a project directory to the config, checked program, and durable state they actually need, then print diagnostics, program output, or a tooling report in the formats that command owns. `check`, `evolve`, `test`, and `data` keep their structured `text`/`json`/`jsonl` reports; `run` keeps `text`/`json`; trace, backup, and restore are text-only. All meaning — parse, type-check, run/test admission, evolution verdicts, store keys, value decoding — is decided downstream in `marrow-project`, `marrow-check`, `marrow-run`, `marrow-json`, and `marrow-store` and only consumed here. `marrow-json` owns shared outbound rendering for run result/error envelopes, run diagnostics, run store-state DTOs, run-fact DTOs, saved keys, data snapshot stamps, check surface ABI descriptors and route manifests, checked surface read and computed-read request DTO decode, generated write request DTO decode, accepted-catalog action argument/result DTOs, computed-read result DTOs, and result rendering; the CLI owns argv, process streams, and command-specific report placement.

Four crates meet here: `marrow-project` owns the `marrow.json` schema,
source/test discovery, and the digest helper; `marrow-check::project_io` owns
project/catalog IO shared by CLI callers; `marrow-json` owns shared outbound
run result/error envelopes, run diagnostics, run store-state DTOs, run-fact
DTOs, saved-key, data-snapshot, surface ABI descriptor, and
surface-route-manifest DTO leaves plus checked surface read/computed-read request
decode; `marrow` is the binary that dispatches argv to one `cmd_*` module and
places command output on stdout or stderr.

## The shared spine

`main::main` installs a broken-pipe panic hook: a `Broken pipe` payload exits 0,
and every other panic defers to the default hook. It then dispatches `argv[1]`
to one command on a worker thread with a large stack (`run_on_worker_stack`).
The parser and runtime recurse over untrusted source on that stack, sized so
their fixed depth limits (`check.nesting_limit`, `run.depth`) always trip before
it overflows. Commands that read an existing project call CLI wrappers in
`main.rs` over shared project IO from `marrow-check::project_io`:

- `load_config` / `load_checked_project` — dir to `ProjectConfig`, then to a `CheckedProgram` bound against an accepted catalog supplied by the caller.
- `native_store_path` / `resolve_store_path` / `open_store_for_inspection` — locate and open the configured store; inspection uses `open_read_only` and then proves the store fully traversable through `verify_readable`, the same walk `recover` runs, so the lightweight render verbs (`stats`/`roots`/`dump`/`get`) are never more permissive than the verifying ones, while write-capable commands opt into the write-open path. Before any open, inspection, `doctor`, `recover`, `serve` (both modes), and `run` run `marrow_check::guard_data_dir`, the shared classifier for a `dataDir` occupied by a non-directory (the same condition `resolve_store_path`'s `create_dir_all` catches on the write path, including `evolve apply`): it reports `config.data_dir` with its directory-creation remedy instead of leaking the open's raw `ENOTDIR` as `store.io`. `run` runs it before the lock-root witness, which opens the store directly, so a committed lock recording active roots cannot reclassify the occupied `dataDir` as a store failure. The presence gate (`marrow_check::tooling::store_path_is_absent`, shared by inspection, lenient `check`, `doctor`, `recover`, `run`, and `evolve apply`) treats only a `NotFound` stat as a first-run absent store: a path occupied by an unopenable file — a symlink loop — is present, not absent, and routes to the typed open failure rather than being blessed as an empty first run. The committed-lock root witness has one bare comparison in `marrow-check` (`verify_store_roots_against_lock`), but no driver calls it directly: the single owner `verify_present_store_lock_roots` (also in `marrow-check`) is its sole caller. Every read-only inspection, `doctor`, `backup`, and `recover` routes through the CLI's thin `verify_lock_roots` wrapper over that owner, and the write-capable `run` and `evolve apply` route through it before establishing a baseline. The discriminator is present-and-damaged versus absent: a **present** store presenting fewer committed roots than the lock records has lost durable identity and fails closed as `store.corruption`; an **absent** store body is the disposable-store case the write paths seed from the committed identity, never corruption — so the read-only inspections, `doctor`, `backup`, and `recover` treat it as a clean first run. A `--backup` mount is self-contained and skips the live-lock witness. A racing reader that opens while a writer holds the redb write lock resolves to `store.locked` at `TreeStore::open`, never a false corruption. `recover` opens write-capable, proves convergence with a fresh read-only reopen, then runs the same owner; an absent store body reports nothing to recover, and a writer that wins the flock or removes the file between those passes surfaces `store.locked` or `store.io`, never a false `store.corruption`.
- `read_committed_lock` — read the committed `marrow.lock` projection as a typed value, failing closed on a corrupt lock (`catalog.lock_corrupt`). `check` uses it to bind first-run identity and to report a stale lock by comparing its source digest against the program's, without opening or repairing the store. Under `--locked` an absent lock over a project that has durable shape to lock — a store carrying a committed catalog, or a recovery-required store file that may carry one — is a fatal `check.lock_missing`, distinct from the present-but-behind `check.stale_lock`; the advisory mode stays silent on absence so a legitimate first run, including a uid-only store stamped before any baseline was published, passes.
- `read_accepted_store_catalog` / `read_accepted_store_catalog_lenient` — the durable-state loaders that bind accepted identity from the live store, the sole write-time authority. The strict form is for commands that inspect or write the store; the lenient form is for read-only `check`, returning a tri-state `AcceptedAuthority` — `Readable(Some)` (binds the committed snapshot and demands a lock under `--locked`), `Readable(None)` (a uid-only store with no committed catalog, which has no durable shape to lock and is a first run), `ExistsButUnreadable` (a crashed/locked/corrupt store file that `check` neither repairs nor fails on but whose presence may carry a committed catalog and so demands a lock under `--locked`), or `Absent` (a first run). A present-but-unreadable store is never collapsed into absent, so a post-crash store that may hold a committed catalog cannot masquerade as a first run and pass an absent lock green; a uid-only `Readable(None)` store carries no committed catalog, so it does not. The lock remains the first-run anchor whenever no readable snapshot binds.
- `reproject_committed_lock` — regenerate `marrow.lock` as a one-way projection of the store's committed snapshot. An authorized write path (evolve apply, baseline freeze) runs it after the store transaction commits, so the command is not done until the re-projected lock is written; it never feeds the lock back into the store. The single lock-write owner (`project_store_lock`) reads the existing committed lock and never regresses it: the projected epoch high-water is the max of the lock's high-water and the snapshot epoch, and the retired-id ledger is the union of the lock's previously-committed tombstones with the snapshot-derived ones (minus any id the snapshot now holds as an active entry). The lock is written atomically through a sibling temp file and an `fsync` + rename, so a torn write can never leave a corrupt lock.
- `establish_store_baseline` — freeze a project's first proposed identity into a write-capable store in one transaction (catalog rows, epoch, engine profile, commit metadata via `marrow_run::evolution::commit_catalog_baseline`), then rebind the program against the now-accepted snapshot. Runs only over an empty store with a pending non-empty proposal; a project past its baseline never churns.
- `client_freshness` / `write_declared_client_if_changed` / `sync_declared_client` — `client_freshness` is the single read-only owner of the declared-client (`config.client`) stale-or-absent verdict: it builds the surface ABI through the `marrow-json` render owner and compares the on-disk header digest against the digest the surface would write, returning a typed `ClientFreshness` (`NotConfigured`, `SurfacelessConfigured`, `AlreadyFresh`, or `Rewritten` for stale-or-absent). `write_declared_client_if_changed` consumes that verdict and renders+writes only on `Rewritten`, so a non-surface edit never churns the file. A live `run`, `serve` startup, and `evolve apply` each call `sync_declared_client` in lockstep with the lock they re-project; a `client` set without any surface emits the `config.client_without_surface` warning and writes nothing, and an isolated dry run skips the write entirely. Read-only `check` never writes the client: it consumes the same `client_freshness` verdict and treats `Rewritten` as a single stale-or-absent condition — a non-fatal advisory by default, a fatal `check.stale_client` under `--locked`. A project with no `client`, or none with a surface to project, raises no client condition.

`marrow run` and `marrow test` use `marrow-run::ProjectSession` for project checking, catalog binding, store admission, and entry invocation. The command modules parse flags, choose hooks, and render the resulting reports. For `run --arg`, `cmd_run.rs` is only the argv adapter: it preserves repeated `name=value` pairs in argv order and passes them to `marrow-run::entry::CheckedEntryCall`, where the checked signature owns textual decoding. For `run --format json`, `cmd_run.rs` captures program output and store stamps, then emits the typed run result or error envelope owned by `marrow-json`. `marrow serve` uses `ProjectSurfaceReadSession` by default, `ProjectSurfaceSession` with `--write`, and validated `surface.route.v1` bindings over the `surface.operation.v1` DTOs to run the dependency-free loopback HTTP profile; the CLI owns the listener, HTTP subset, process lifetime, mode selection, optional exact loopback CORS origin, and strict request routing, while `marrow-json` owns ABI-derived operation catalog construction, route/ABI validation, JSON decode/execute/render for read, computed-read, create, update, delete, and action operations. `marrow client typescript` uses the check-style project path that binds first-run identity from the committed `marrow.lock` projection, then asks `marrow-json` to render the typed TypeScript surface client from the same ABI and route manifest.

Stream separation is load-bearing: in text mode a program's own `print` output owns stdout; run tooling reports such as trace and dry-run plans go to stderr, so a stdout consumer never sees interleaving. `run --format json` switches to the envelope egress regime: stdout is a single result envelope containing captured program output and any renderable return value, while runtime faults still render on stderr. Trace is text-only and streams directly. `marrow test --format json|jsonl` owns stdout for its structured test-result report, with text trace output kept on stderr. Exit codes are 0 success, 1 failure, 2 usage.
Structured reports that include a `project` field render the canonical absolute project directory through `project_json_path`.

## Command families

| Command | Module | Behavior |
|---|---|---|
| `init` | `cmd_init.rs` | Creates the quickstart scaffold in a new target directory after validating the final path component as one module-name segment. Bare `init` is store-only; `--client`/`-c` threads a `client: bool` through the single scaffold owner to also emit a `surface` over `^books` and a `client` output line. |
| `check` | `cmd_check.rs` | Type-checks a project directory containing `marrow.json`; JSON output includes checker entry footprints, serialized surface ABI descriptors, and descriptor-derived `surface.route.v1` route manifests for successful checks. |
| `doctor` | `cmd_doctor.rs` | Aggregates read-only operator triage findings with exact next commands, including store-open failures, catalog validation, fence/stamp classification, engine profile, and a capped integrity sample. |
| `run` | `cmd_run.rs` | Parses run flags and repeated `--arg name=value` pairs, opens a `ProjectSession`, emits session notices, invokes the selected entry under a plain/trace/dry-run hook, and renders text output, JSON envelopes, or dry-run reports. |
| `test` | `cmd_test.rs` | Opens a test `ProjectSession`, filters its public zero-param test cases by qualified name substring, invokes each selected test, and renders pass/fail/error reports. |
| `client typescript` | `cmd_client.rs`, `cmd_client/typescript.rs` | Parses the client generator command and optional `--out <path>`, checks the project against the committed `marrow.lock` without opening the store, reports ordinary check diagnostics on failure, warns on stderr when the lock is behind the source (`check.stale_lock`), and on success writes the self-contained typed TypeScript surface client to `--out` (relative resolved against the cwd, absolute as given; resolved path reported on stderr), or with no `--out` refreshes the declared `marrow.json` `client` path write-if-changed through the shared declared-client owner that `run`/`serve`/`evolve` use, or prints to stdout when no `client` is declared. |
| `serve` | `cmd_serve.rs`, `cmd_serve/cors.rs` | Parses the loopback bind target, optional `--write` mode, optional exact loopback `--cors-origin`, and optional `--watch`, opens `ProjectSurfaceReadSession` or `ProjectSurfaceSession` and holds it for the server's whole process lifetime so the native store lock makes the server the owning process (a racing writer or read-only inspection is refused `store.locked`), builds ABI-derived operation catalog and validated route bindings from `surface.route.v1`, filters allowed routes by mode, accepts one bounded JSON `POST` per connection plus matching CORS preflight when configured, validates route/body operation agreement, dispatches each request through the held session, and returns sanitized JSON success/error envelopes. Under `--watch` the accept loop polls a source-mtime signature over the source-root `.mw` files between accepts and, on a change, releases the held session — and its lock, since a re-check re-opens the store — re-checks, rewrites the declared client write-if-changed, and re-acquires a fresh session and routes, logging a tooling error on a failed re-check. |
| `fmt` | `cmd_fmt.rs` | Formats one file to stdout, or `--check`/`--write` over source roots; refuses stdin, a bare dir with no mode, and any mode whose output would reduce retained comments (`fmt.comment_loss`), so stdout, `--check`, and `--write` agree on losslessness. |
| `data <roots\|stats\|dump\|integrity\|recover\|get>` | `cmd_data.rs`, `cmd_data/` | Store inspection plus explicit recovery; read-only views either pin one live-store `ReadSnapshot` or mount `--backup` into memory, while `recover` performs only a write-capable store open. |
| `evolve <preview\|apply>` | `cmd_evolve/` | Read-only preview vs managed-write apply; preview can render parseable evolve scaffolds or derive the witness from `--from-backup`, and apply gates Retire-bearing witnesses on a recovery point before committing data plus catalog rows atomically. |
| `backup` / `restore` | `cmd_backup.rs`, `cmd_restore.rs`, `backup/` | Read-only archive write through a shared atomic artifact helper over a pinned snapshot, carrying the accepted-catalog rows in a typed section; transactional all-or-nothing replay of catalog rows and data into an empty native store, or an explicitly counted replace target, which then runs without re-running evolution. Restore re-establishes durable identity, so it re-projects the committed `marrow.lock` from the restored snapshot through `reproject_committed_lock`, leaving an immediate `check --locked` clean. |

## Load-bearing invariants

- **Path containment.** Every project-relative path (source roots, `dataDir`, tests, the optional `client` output) is rejected if empty, absolute, or containing `..`, because each is later `Path::join`ed onto the project root. Test paths additionally reject glob metacharacters, so discovery is plain file-or-directory selection rather than pattern matching. `parse_config` double-parses (raw `Value` then typed `RawConfig`) to catch non-object roots and unknown-key spans.
- **Run admission.** `ProjectSession::open` owns run admission. A clean project
  with a pending durable identity has its baseline frozen into the store the
  first time a real `run` opens it write-capable; a memory-backed project with a
  durable surface refuses with `run.durable_store_required`, and a plain script
  runs over memory.
- **Run drift handling.** On native schema drift, a zero-record-mutation change
  is auto-applied through the production apply path, which publishes the
  advanced catalog snapshot in the apply transaction; the session then
  re-checks and re-fences. Any backfill/transform/destructive change fences
  naming `evolve apply` instead. The redb file lock forces real apply paths to
  drop the first handle before reopening. A store holding records under no
  accepted catalog is refused as populated-but-unstamped.
- **Dry-run classification.** `run --dry-run` takes the same checked path but
  opens native stores read-only for classification: it reports would-freeze,
  would-apply, or would-fence as tooling content, does not freeze or auto-apply,
  and does not execute the entry when the fence would not pass.
- **Arguments and envelopes.** Text `SessionEntry` calls carry the selected
  entry name plus raw text argument pairs into `CheckedEntryCall`, where the
  checked signature decodes scalars, enums, scalar/enum sequences, and
  single-key identities or rejects unsupported surfaces with
  `run.entry_argument`. Protocol `SessionEntry` calls carry a
  `marrow-check` `entry.invoke.v1` descriptor identity plus typed argument
  values; `marrow-run` admits that identity against the current checked runtime
  by comparing the callable ABI tag and read-only checked-program context, then
  decodes only canonical scalar, enum-member, identity, and supported sequence
  values before execution. Descriptor tags bind entry name, parameter shapes,
  return shape, accepted catalog identities, and return presence, not function
  bodies or source-spelled return type names; body-only implementation changes
  under the same signature do not stale a protocol invocation.
  `cmd_run.rs` adapts argv, captures program stdout for `--format json`, records
  session-owned store stamps, and serializes the typed run DTOs from
  `marrow-json`. Resource and local-tree returns remain outside that return
  surface and fault as `run.entry_surface`. The envelope emits `committed` only
  when the invocation advanced the store commit.
- **Restore is all-or-nothing.** The whole replay runs in one transaction; any checksum/framing/verify failure rolls the target back to its prior state. Restore targets are empty-only by default; counted replace mode first confirms the live record (entity) count from `--replace --count N`, then clears and replays inside the restore transaction. Restore carries data cells only (indexes are rebuilt), replays bytes verbatim only when `EngineDescriptor` matches exactly, and proves the data compiles against the schema via the `verify` closure before commit. The engine-match gate fires only after the integrity checksum verifies the manifest, so a byte-tampered engine block is reported as corruption rather than a foreign-engine recompile. Raw byte validity is never enough.
- **One record count, one owner.** A user-facing "record" is a saved entity (identity tuple); a "cell" is a stored `(path, value)` pair. `backup::count_live_entities` is the single owner of the record count across the inspection surfaces: `data stats records:`, the backup `record(s)` line, and the `restore --replace --count N` guard all report it. It descends each store at its declared identity arity, so it agrees with the data inventory and skips a malformed over- or under-length node. Evolution counts entities through `marrow-store`'s `data_record_count`, which sums distinct data-family identities; the two agree on well-formed data and differ only in that `data_record_count` tolerates a malformed node the inventory skips. `data stats cells:` and `data integrity` count field cells; the backup manifest's `record_count` is an internal physical cell-frame count for replay, never user-facing.
- **Backup-backed reads are non-activating.** `data roots|stats|dump|integrity|get --backup` loads the current config and source, validates the backup prologue and carried catalog through the restore contract, replays the archive into `TreeStore::memory`, verifies the mounted data against the checked schema, and then inspects that memory store. The shared artifact open (`read_backup_artifact`, also used by `restore` and `evolve preview --from-backup`) first guards that the path resolves to a regular file: a FIFO, socket, or device is refused as `restore.format_version` before the blocking `File::open`, which a writer-less FIFO would otherwise hang on forever. It does not open the configured native store, take the store lock, regenerate `marrow.lock`, or write durable state. `evolve preview --from-backup` uses the same in-memory data mount for its witness but still compares the project's accepted identity with the backup identity, so a project that has advanced past the backup reports the typed restore drift refusal instead of previewing across catalog histories.
- **Evolve apply.** `evolve apply` first runs the shared lock-root witness (`verify_lock_roots`) so a **present** store that lost the roots its committed `marrow.lock` records — rolled back, partially dropped, or crashed mid-creation — fails closed as `store.corruption` rather than being re-baselined over the loss; an **absent** store body is the disposable-store case, so apply seeds and baselines an empty store from the committed identity through the shared `marrow_run::seed_absent_store_from_committed_lock` owner and announces the seed loudly (`SeededFromCommittedLock`), and a genuine first apply records no active root in the lock, so the fresh-baseline path runs. It then opens the writable native store and stamps its durable UID before anything else, so the `(store_uid, commit_id)` identity is in place before any baseline or catalog is committed and no crash window can leave a durably committed accepted catalog with an absent UID, binds accepted identity from the live store snapshot, the sole write-time authority, freezes a pending baseline into the store, and previews the witness. A Retire-bearing witness requires `--backup <path>` or `--no-backup`; the backup path must not resolve under managed project paths (`marrow.json`, `marrow.lock`, source roots, test paths, or the native data directory/store file), then uses the shared typed backup artifact writer and validates the archive before apply stages evolution work. A desynced apply fails closed: when the committed `marrow.lock` records an epoch high-water this single activation cannot reach (the target epoch is below the lock's high-water), the apply refuses with the typed `run.store_behind` fence rather than mint a fresh activation below a teammate's already-committed epoch. The fence carries an apply-specific remedy: it tells the operator to reconcile the local store with the team's up-to-date store (pull or rebuild the store that matches the committed lock), never to re-run the apply that just refused. An exact catch-up to the lock's high-water is allowed. The apply advances the catalog snapshot, the epoch, and the data in one store transaction; after commit, the CLI regenerates `marrow.lock` as a one-way projection of the committed store snapshot, taking the monotonic max of the lock high-water and unioning its retired-id ledger so the projection never regresses an ahead lock.
- **Render-only prose.** Integrity, evolve, and restore code assert stable dotted codes and typed verdicts/problems; diagnostic prose is never matched semantically.
