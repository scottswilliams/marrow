# Store

`marrow-store` is the bottom of the stack: a typed tree-cell storage contract (`TreeStore`) over a physical ordered-byte key/value grammar, with two interchangeable engines behind a private `Backend` trait. It assigns no language meaning тАФ callers pass opaque `CatalogId`s and typed `SavedKey`/`Scalar` values; this crate owns only the durable byte forms and the rules that keep them stable across reopen.

## The one big idea

Keys are order-preserving; values are not. `SavedKey` encodes scalars so byte-lexicographic order equals Marrow's typed key order (sign-flipped big-endian numbers/dates/durations/instants, `0x00`-escaped strings/bytes), so every range scan runs without a comparator. Values use a separate canonical-but-unordered codec, because the store sorts by tree-cell key, not by value. Both forms are round-trip-exact: decoders re-encode and require byte-identical output, so non-canonical spellings (`+1`, `01`, `1.50`, `-0`, trailing-zero fractions) are rejected rather than normalized.

## Parts

- **Engine contract** тАФ the private `Backend` trait (read/read_prefix/write/delete/scan/begin/commit/rollback/snapshot) plus `ScanPage`, `ValuePrefix`, and typed `StoreError`. Two engines implement it: `MemStore` (BTreeMap) and `RedbStore` (persistent). A shared `conformance` suite runs 22 laws against both.
- **Key grammar** тАФ `cell` defines the v0 physical key layout (placement prefix, profile, family, store id, identity, segments); `key` defines the order-preserving scalar codec used in every record-key and index-key position.
- **Value forms** тАФ `value` (canonical scalar codec, calendar years 0001тАУ9999) and `decimal` (exact base-10, 34-digit/34-place envelope, half-to-even division and scale rounding).
- **Public facade** тАФ `TreeStore` wraps a boxed `Backend` and exposes every typed write/read/navigation/transaction/snapshot/backup call other crates use.
- **Durable stamp metadata** тАФ `metadata` (`CommitMetadata`, `EngineProfile`, `StoreUid`, and the source digest the activation fence binds).
- **Catalog table** тАФ `catalog` persists the accepted
  `marrow_catalog::CatalogMetadata` as a header row plus one row per entry in
  its own physical family (`FAMILY_CATALOG`), written in the caller's
  transaction and invisible to data/index/meta access. A read verifies the stored
  header's canonical digest against the decoded rows and rejects any mismatch as
  corruption.
- **Backup** тАФ `backup` streams data-family record presence cells, keyed group-entry path nodes, and value cells; index cells are rebuilt, and commit metadata and the per-root structural digests are restamped on restore (the digests recomputed from the replayed cells), never archived.

## Invariants worth knowing before you touch bytes

- `0x00` is the only structural separator. Strings/bytes/ids escape it (`0x00` тЖТ `0x00 0x01`) and terminate with `0x00 0x00`; a compile-time assert keeps the date tag adjacent to int (`KEY_DATE == KEY_INT + 1`), and `KEY_STR` is fixed at `0x07`.
- Transactions are atomic across the whole staged plan. A mid-plan fault rolls the entire bracket back with no surviving write and no metadata stamp. `MemStore` snapshots once per flat transaction; `RedbStore` runs one long-lived write transaction and aborts it on rollback.
- Native redb commits pin immediate durability and keep redb's one-phase commit posture. Fresh store creation fsyncs the containing directory after the first format-stamp commit.
- A pinned read snapshot is mutually exclusive with writes on the same handle (offenders get `store.transaction`) and is non-reentrant, so a multi-page backup sees one coherent version.
- Scans are always bounded and resumable. A truncated page with no resume key is treated as corrupt scan state (`store.cursor`), never as the end.
- Catalog snapshot integrity is independent of declaration order. New headers
  store a digest over entries sorted by kind tag, path, stable ID, aliases,
  lifecycle tag, accepted store-key shape, accepted store-index shape, accepted
  structural signature, and the applied-transform mark. Reads accept only that
  canonical digest.
- Corruption is fail-closed and typed: malformed keys, value frames, metadata, or backup frames decode to `StoreError::Corruption`, never partial values. Decoders enforce exact-length consumption and bounded list counts.
- On-disk format is version-gated with no automatic conversion: a mismatched `FORMAT_VERSION` is refused, a non-empty redb file with no `marrow.meta` table is corruption, and a read/write lock conflict is `store.locked`.
- Native access fails closed, never crashes: opens, reads, writes, and the database close all run under one panic backstop (`catch_store_panic`, reached via `catch_open` for opens and `catch_traversal` for every `RedbStore` read/get/scan, write/delete transaction, and the `Drop` that closes the handle). redb reads btree pages lazily, so damage below the table roots opens cleanly and only panics тАФ a layout assertion or slice-index `unreachable!()` тАФ when a read traverses the tree, a write transaction descends it, or the close flushes redb's allocator and region bitmaps; `catch_traversal` maps that panic to `store.corruption` for every access shape (point get, forward and reverse scan, seek, snapshot pin, insert/remove, close) instead of aborting the process. Navigation that seeks the next/previous child from a prior key (record and data-path descents, index walks) holds a strict-monotonic-progress guard: a child that does not advance past the cursor is backend damage, surfaced as `store.corruption` rather than spun in an unbounded loop. The same guard sits one level lower, on every paged scan loop that resumes from a page's final key (the family walk `data integrity`/`stats`/`dump` drive and the record/child seek loops): a resume key that does not advance past the cursor it was seeded with is damage, failed closed instead of re-querying the same cursor forever. The bounded-batch delete resumes each batch strictly past the prior one so a re-entered damaged page is seeked over, not looped on. `recover` attempts a write-capable repair and then proves it converged: it opens write-capable and walks the tree via `TreeStore::verify_readable` (which decodes and value-validates every data cell, rejects an out-of-order identity, descends each `(store, record arity)` through the same seek-driven `for_each_record` navigation `data integrity` uses, so a damaged page that hides off the linear scan but on a record re-seek surfaces `store.corruption`, and re-derives each root's structural digest from a full data-family scan and cross-checks it against the durable per-root digest each commit stamps, so a page that silently drops a cell, truncates a record range, or rewrites a stored value without a structural fault fails closed too), drops that handle so redb persists what the replay produced, and finally re-opens read-only and re-walks. The derived index family is verified by `verify_index_readable` (a linear structural decode of every index cell in encoded order, then a seek-driven re-descent of every `(index, key arity)` through `index_first_child`/`index_next_child`) and by a schema-driven completeness cross-check: both structural passes navigate the index btree, so a damaged page that silently truncates an index range hides the dropped entries from each, and the data records are the independent oracle. The cross-check (`verify_store_completeness` in `marrow-check`, which also runs the structural-digest check) derives the entry count each declared index owes from the data records тАФ every record whose indexed columns are all populated publishes one entry, the rule the runtime write path applies тАФ and rejects any index whose enumeration disagrees. The committed-lock root witness that rejects a store presenting fewer accepted roots than its `marrow.lock` recorded тАФ catching a rollback that drops the digests together with the data тАФ is a separate cross-check the CLI runs through its race-aware owner, not part of the completeness gate. `data integrity` runs both; `recover` runs the completeness gate when the project checks (broken source still gets the store-level repair) and always runs the lock-root witness; `backup` runs them before archiving, since its data-only stream rebuilds the index on restore and would otherwise mask the under-read. Allocator- and slot-region damage can replay into a handle that reads cleanly yet leaves the file still demanding recovery on the next open; the fresh read-only re-open is the convergence oracle, so recover reports success only when the next read would succeed and otherwise reports `store.corruption` rather than a false repair. Every open first guards that the store path resolves to a regular file: a FIFO, socket, or device is rejected as `store.corruption` before redb's `O_RDWR` open, which a writer-less FIFO would otherwise block in forever. It then guards the super-header (`guard_superblock_header`): redb maps a page by reading the byte span its page number names, so a committed btree root clobbered toward a huge region/index/order would make the open allocate gigabytes before any data is read. Each non-null root in both commit slots is checked against the actual file length and a root whose page span lies past the file is rejected as `store.corruption`, bounding the open regardless of which header field is corrupt. A read-only or existing-store open also tolerates the brief creation race in which a first-run writer has made the file but not yet committed its header under the lock: a torn-body corruption reported while the file is still zero-length is retried over a short budget, so a racing reader resolves to the healthy store or the lock rather than a false `store.corruption`, while a settled writer-free empty file stays empty across the budget and is still rejected as corruption. redb open errors map by damage (`map_open_error`): a torn body to corruption, an unclean-shutdown repair-needed file to the typed `store.recovery_required` (a write-capable open attempts the replay and reports whether the store opened), a read/write holder conflict to `store.locked`, a denied open to the path-bearing `store.permission_denied`, everything else to `store.io`. A read-only open is otherwise more permissive than the write-capable open `recover` and `run` use: it loads the last clean commit's recorded allocator state and reads through its roots, so god-block or commit-tracker damage that the write path rejects opens cleanly and a content walk reads past it. So `open_read_only` follows the read-only open with `verify_committed_recoverable`, which replays redb's write-capable open detection over the committed bytes through a `RecoveryProbeBackend` that reads the real file and discards every write тАФ surfacing the same `store.recovery_required` or `store.corruption` the write path reports, without touching the file or taking its lock, so the inspection family is never more permissive than `recover`.
