# check/catalog — stable durable identity

After type analysis builds a `CheckedProgram`, this layer gives every durable declaration (resource, store, store index, enum, enum member, resource member) an opaque stable id that survives renames and reshapes, and derives the source digests the store stamps and the activation fence enforces. The accepted reference is a caller-supplied input: `bind_catalog` takes an `Option<&CatalogMetadata>` snapshot and an `Option<&CatalogLock>` projection the analysis API threads in. The CLI binds the live store snapshot when one is present and falls back to the committed `marrow.lock` for first-run adoption, so the checker itself never reads durable identity from disk; it reconciles source against that reference and *proposes* an advanced catalog. It is read-only — the checker neither reads nor writes durable identity. A separate `rejected_surface` pass rejects v0.1-forbidden saved-traversal calls.

## The big idea

Identity is path-independent. A stable id is a random 128-bit `cat_<32hex>` minted from OS entropy, not derived from the source path — so renaming an entity carries its id forward instead of inventing a new one, and branch-parallel allocation cannot collide the way a counter would. Reconciliation matches source entries to accepted entries by `(kind, path)`, relocates renamed entries (recording the old path as an alias), reserves retired entries, drops any active entry the current source no longer declares and no intent relocated or retired (a bare removal: a resource, store, field, enum member, or derived index), and mints fresh ids only for genuinely new ones. Dropping a bare removal keeps an in-place run's projection identical to a fresh reseed of the same source — the same entries, with a re-add minting fresh identity — and never authorizes data loss: activation discharge fences a removed entity that still holds records, read against the accepted snapshot, before the proposal can commit. Structural change is detected from recorded signatures, not source spelling, so a re-key, a group↔keyed-group reshape, or a value retype advances the proposal even when names are unchanged.

## Parts

- **Snapshot model** (`crates/marrow-catalog`): the accepted-snapshot types `bind_catalog` consumes and proposes — `CatalogMetadata`/`CatalogEntry`, epoch/digest validation, store-index shape recording, structural-signature decode.
- **Binding** (`catalog/mod.rs`): `bind_catalog` orchestrates carry-forward, rename relocation, retire reservation, and fresh-id mint, then records store-key, store-index, and member signatures, builds the proposal, and binds accepted-only ids onto facts.
- **Source digest** (`catalog/source_digest.rs`): two sha256 digests over the canonical schema structure — never rendered source text. Each durable entity contributes a record of its module-qualified path and its shape by source identity (member value type and key shape by type name, store identity-key types and target resource, index uniqueness and key columns, enum member by name chain and ordinal, const by canonical value expression). Records are hashed as a set, so a declaration reorder is stable, while a member's ordinal makes a member reorder move it. The shape digest excludes the `evolve` block (a consumed block is deletable); the evolution digest adds each bound default value and transform body.
- **Id allocator** (`catalog/stable_id.rs`): opaque random id minting, re-rolling against every recorded id of any lifecycle so retired ids are never reissued.
- **Rejected surface** (`rejected_surface.rs`): walks function/const/transform bodies and rejects forbidden saved-traversal methods.

## Contracts that bite

- **Two id maps by design.** `ids` (accepted-only) binds onto live facts; `leaf_token_ids` / the proposal-inclusive map covers freshly-minted and renamed referents but is used *only* for leaf-token resolution and never binds onto facts — a proposal-only identity cannot leak into the program.
- **Proposal advances only on real change.** Exact source-vs-accepted match returns `None`. Accepted and proposed catalogs are validated at check time, so an id collision or missing store-index declaration shape fails closed before binding. Backfilling a store-key or member signature for an entry that never had one recorded is not change; store-index entries must already carry their accepted declaration shape.
- **Rename is identity-preserving and injective.** A rename relocates the accepted entry to its new path and keeps it Active; resolution is rejected if the source path is still declared, the target already names a live entity, or no active accepted entry backs the source path. Renaming an ancestor cascades to its whole accepted subtree, since a leaf's saved identity is keyed on its full ancestor path: renaming an enum category carries its descendant enum members, and renaming a resource carries its resource members, so stored descendant values stay valid without a per-descendant rename (`cascade_descendant_kind`/`cascade_subtree_renames`). A retire only reserves once the source declaration is gone.
- **A pending entity is a Warning, not a failure.** Source absent from the accepted catalog and not renamed stays informational; identity is frozen only when run/apply commits.
- **Lock adoption seeds first-run identity, never overrides a store.** When no accepted store snapshot is present, a committed `marrow.lock` adopts identity for the current source by matching each source `(kind, path)` to a lock entry, carrying that entry's rename aliases and consumed-transform mark forward onto the adopted row and seeding the allocator from the lock's append-only ledger and `epoch_high_water` so a retired id is never reissued and the epoch never regresses. Adoption also *materializes* each ledger tombstone — which carries the retired `(kind, path)` alongside its id — as a Reserved entry in the proposal, so a fresh checkout that re-seeds a wiped store from the lock alone commits the reserved rows back into the store: the projection re-derives the same ledger and re-declaring a retired path fails closed identically to the live-store path, with the never-reuse defense surviving store loss. A pending or already-applied rename or retire block resolves against the lock-recorded identity exactly as it does against a present store: a pending rename carries the committed entry forward under its new path with the old spelling as an alias, a pending retire reserves the committed entry at its retired path, and an already-applied block whose old spelling is an adopted alias or materialized Reserved row is recognized as recorded — none a `check.evolve_target` failure; only an intent naming nothing the lock or source reconstructs fails closed. A consumed `evolve transform` is likewise recognized through the discharge mark the adopted row restores, so a kept transform block does not re-fire against the empty seed, while a transform the lock never recorded as applied stays a genuine pending obligation. A materialized Reserved entry is identity-only and records no live shape, so the shape-presence invariants apply to active entries alone — a retired store index reseeds as a shapeless Reserved row rather than tripping the accepted-shape check. Adoption fails closed on a tombstone reissue. The advance rule that mints the next epoch saturates rather than overflowing, and the lock codec rejects an `epoch_high_water` no real change could advance past (within one of `u64::MAX`) as `catalog.lock_corrupt` at decode, so the version line stays strictly monotone. A valid store snapshot always takes precedence; the lock is a first-run anchor only. When the lock adopts the source *cleanly* it is the accepted reference itself: the binding carries the same canonical `CatalogMetadata` (epoch, order-independent digest, entries) a present store snapshot would, so a checkout binds the identical accepted digest — and the read-only context digest that folds it in — whether it reads identity from the committed lock or from a present (or momentarily writer-locked) store. The surface computed-read operation tag folds that digest, so it stays stable across those states. When the source has *drifted* from the lock instead, the first-run proposal is a pending change, so it advances one epoch past the high-water exactly as a present store discharging that change would (and records a pending shape-neutral transform's consuming mark the same way); freezing it AT the high-water would fold the delta into the committed epoch and under-advance a fresh checkout relative to the present-store path on identical committed inputs.
- **The digest reads structure, not text.** It is computed from the typed schema — each entity's path and its shape by source identity (type names, key types, uniqueness, enum member name chains and ordinals, const values), never from rendered declaration text. So a reformat, blank-line change, or comment edit cannot drift it, while every structural edit does; a declaration reorder is stable (records are a set) but a member reorder is not (a member's ordinal is part of its record, matching the store's restamp-on-reorder contract). Type identity is the source spelling, not a minted catalog id, so the digest reproduces for the same source on any machine.
