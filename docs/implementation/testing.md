# Testing implementation

Marrow uses several evidence layers. A green layer proves only its own scope;
unit tests do not substitute for production-pipeline or crash tests.

| Layer | Purpose |
|---|---|
| Unit tests | Local parser, checker, codec, runtime, and store invariants. |
| Source-driven integration tests | Complete projects through check, run, test, and tool commands. |
| Language conformance projects | Shared public behavior expressed in `.mw`. |
| Store conformance laws | Identical logical traces over memory and redb. |
| Crash and corruption tests | Fail-closed open, commit, recovery, and bounded hostile input. |
| Boundary tests | CLI and structured transport mapping. |
| Architecture/absence tests | Dependency direction, private APIs, generated drift, and deleted patterns. |

Tests should assert typed codes, spans, values, facts, store effects, witnesses,
and receipts rather than prose messages. CLI tests remain thin when the same
behavior can be exercised below process rendering.

## Fixtures

`fixtures/v01/conformance/` contains complete Marrow test projects.
`fixtures/v01/evolution/` contains historical and target source variants.
Runtime fixtures cover resources, indexes, identities, and traversal. A fixture
must use the production parser/checker/runtime path appropriate to the behavior
it claims.

## Documentation

Every current `mw` fence is a complete module checked through the production
project pipeline. The canonical sample is also executed over memory and native
storage. Workflow guides should have source-driven fixtures for their complete
command journeys; transcript prose alone is not evidence.

`docs/error-codes.md` is generated byte-exactly from `marrow-codes`. The
implementation crate table, documentation tree, local links, future/current
authority boundaries, and removed legacy paths have explicit drift gates.

## Gate order

Run focused tests first, followed by affected integration tests, full workspace
tests, formatter, Clippy with warnings denied, zero-unsafe checks, generated
artifact drift, and documentation link/example checks. Cargo commands on the
shared build host always use the lane's explicit external `CARGO_TARGET_DIR`.
