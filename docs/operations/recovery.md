# Recovery

Recovery begins by preserving process ownership and classifying the failure.
Do not run a write-capable repair while another process owns the native store.

## First Response

1. Stop the current writer and any long-running read-only inspection.
2. Run `marrow doctor <projectdir>`.
3. Follow the typed code rather than matching message prose.

| Finding | Meaning and next action |
|---|---|
| `doctor.store_locked` | Another process owns a conflicting handle. Stop it and retry the diagnosis. |
| `doctor.store_recovery_required` | The native engine requires a write-capable open. Run `marrow data recover`. |
| `doctor.store_unavailable` | Store open or metadata failed. Inspect `data.underlying_code` for `store.corruption`, `store.io`, or another `store.*` cause. |
| `doctor.integrity_sample_failed` | The bounded modeled-data sample failed. Run `marrow data integrity` for `data.*` findings. |
| `doctor.lock_corrupt`, `doctor.stale_lock`, or another lock finding | Reconcile the lock separately as described below; do not use it to overwrite a valid store. |

`doctor` uses a bounded integrity sample. A clean sample is not a complete scan;
run `marrow data integrity` before returning a repaired store to use.

## Recover An Unclean Open

```text
marrow data recover [--format text|json|jsonl] <projectdir>
```

Recover reads `marrow.json`, attempts a best-effort `marrow.lock` read, locates
an existing native store, and performs the backend's write-capable open/replay.
An unreadable or corrupt lock does not block this physical operation, so rerun
`doctor` afterward to diagnose it. Recover then traverses the store, closes it,
reopens it read-only, and traverses again. A store that still requires recovery
or cannot be read after that fresh open is reported as corrupt rather than as a
successful repair.

Source checking is deliberately not a prerequisite for physical recovery. The
command silently attempts to check source; failure does not block the store
open. When source checks successfully, recover also verifies schema-driven data
and index completeness before reporting success.

A missing store is reported as nothing to recover and is not created. A foreign
redb file, an empty non-Marrow file, lost committed roots in a present store, or
damage beyond replay remains `store.corruption`.

After a successful recover:

```sh
marrow doctor ./project
marrow data integrity ./project
marrow backup ./project ./post-recovery.mwbackup
```

The final backup is optional operational policy, but it establishes a validated
typed artifact after recovery.

## Modeled Data Repair

`data recover` repairs only the physical open/replay path. It does not insert
required fields, rewrite invalid values, resolve dangling identities, accept
source evolution, or delete orphaned managed cells.

Use `marrow data integrity` to obtain typed findings. Apply supported source
changes through `evolve preview` and `evolve apply`. For application-specific
data corrections, write checked Marrow maintenance code and invoke it with
`marrow run --maintenance`; run integrity again before normal use.

## Restore From A Backup

If a store cannot be made readable, restore a known-good typed archive into an
empty native target. Preserve the failed file outside the active data path if it
is needed for investigation; do not copy its raw bytes into the replacement.

For a healthy but unwanted nonempty target, counted replace is available:

```sh
marrow restore --replace --count N ./project ./known-good.mwbackup
```

Replace validates the live target against its accepted shape and counts it. It is
not a bypass around a corrupt target, and its preflight does not replace the
dangling-reference pass in `marrow data integrity`. The
[Backup And Restore Reference](../tools/backup-and-restore.md) defines
validation, transaction, and success-output behavior.

## Lock Recovery

The live store, when valid, owns accepted durable identity. A stale lock is
regenerated by a successful write-capable `run` or `evolve apply`. A syntactically
corrupt `marrow.lock` blocks those paths; after confirming through `doctor` that
the live store is authoritative, remove the corrupt lock and rerun the
write-capable command so it can project a replacement.

An absent store body with a committed lock is disposable-state recovery: the
next write-capable run or evolution apply can seed an empty store from that
identity. A present store missing committed roots is data loss and fails closed;
deleting the store file would discard evidence and must not be presented as a
repair.
