## 3. External Precedents And Counter-Precedents With Source Links

Rust is the strongest precedent for Marrow's "make illegal states hard to express" direction. The Rust Book frames enums as types that enumerate possible variants and highlights `Option` as the ordinary way to model "something or nothing" in the type system; it also uses `Result<T, E>` rather than exceptions for recoverable errors ([Rust enums and pattern matching](https://doc.rust-lang.org/book/ch06-00-enums.html), [Rust error handling](https://doc.rust-lang.org/book/ch09-00-error-handling.html)). Marrow's absence model is stricter and more storage-aware than Rust's `Option`: absence is a path/property fact, not a first-class optional value. That is defensible for saved trees, but less generally composable than Rust-style `Option` values.

Swift and Kotlin are good counter-precedents for user ergonomics. Both put nullability/optionality in the type surface and provide chaining/defaulting syntax; Swift optional chaining returns an optional result, and Kotlin distinguishes nullable and non-nullable types with safe-call and Elvis operators ([Swift optional chaining](https://docs.swift.org/swift-book/documentation/the-swift-programming-language/optionalchaining/), [Kotlin null safety](https://kotlinlang.org/docs/null-safety.html)). Marrow's `?.` and `??` are familiar, but their domain is narrower: they operate on sparse saved paths, not arbitrary maybe values. This is teachable if described as "presence of data at an address", not as "Marrow's optional type".

TypeScript is both precedent and warning. Its control-flow narrowing uses ordinary runtime checks to refine types ([TypeScript narrowing](https://www.typescriptlang.org/docs/handbook/2/narrowing.html)), which is close to Marrow's presence proofs. But TypeScript also shows how easy it is for permissive syntax and truthiness-like shortcuts to accrete confusing edge cases. Marrow's refusal to add truthiness and wildcard enum arms is a strength.

F# is the cleanest precedent for discriminated unions and exhaustive pattern matching ([F# discriminated unions](https://learn.microsoft.com/en-us/dotnet/fsharp/language-reference/discriminated-unions)). Marrow's enum hierarchy has part of that feel, especially with exhaustive `match`, but it is not a full algebraic data type system: enum category nodes, stable catalog IDs, and `is` subtree checks optimize for durable classification rather than local modeling expressiveness. That is a good database-language choice, but it should not be oversold as general ADTs.

SQL and PostgreSQL are counter-precedents on query ergonomics. PostgreSQL has an optimizer that generates alternative scan/join plans and considers indexes where predicates match them ([PostgreSQL planner/optimizer](https://www.postgresql.org/docs/current/planner-optimizer.html)). It also teaches transactions as familiar database units with rollback and savepoints ([PostgreSQL transactions](https://www.postgresql.org/docs/current/tutorial-transactions.html)). Marrow intentionally rejects a hidden optimizer and makes access path part of the source program. This is excellent for predictability and a small v0.1 implementation, but it trades away decades of database ergonomics for ad hoc and analytic querying.

Datalog and Souffle are useful counter-precedents for querying derived facts. Souffle declares typed relations and can auto-select indexes for primitive searches ([Souffle relations](https://souffle-lang.github.io/relations)). Datalog's selling point is declarative "what, not how" querying. Marrow deliberately chooses "how, but checked and source-visible" for saved traversal. That is coherent, but it means Marrow should not drift into an accidental half-Datalog through builtins, helper traversals, or hidden scans.

Datomic is the sharpest data-model counter-precedent. It treats the database as immutable facts accrued through declarative transactions and makes history a first-class property of the database ([Datomic transaction model](https://docs.datomic.com/transactions/model.html)). Marrow's mutable tree paths, exact replacement, and manual history layers are much easier to compile into a small local engine, but less naturally audit-oriented. If Marrow's identity/history examples read like a hand-rolled Datomic, users will ask why transaction history is not a primitive. The answer should be explicit: Marrow is a typed saved-program language, not a temporal fact database.

EdgeDB/Gel and Prisma are useful schema/query/migration counter-precedents. Gel presents EdgeQL as a SQL successor and pairs schema, types, migrations, and a TypeScript query builder ([Gel EdgeQL](https://docs.geldata.com/learn/edgeql), [Gel migrations](https://docs.geldata.com/reference/datamodel/migrations)). Prisma separates an ORM schema model, relations, and migrations from database tables and foreign keys ([Prisma relations](https://www.prisma.io/docs/orm/prisma-schema/data-model/relations), [Prisma Migrate](https://www.prisma.io/docs/orm/prisma-migrate)). Marrow's refusal to have a separate schema/query/migration language is a principled simplification, but it also means the single language must carry examples, diagnostics, and tooling quality that those ecosystems distribute across multiple tools.

Embedded DSLs, especially LINQ, show the other obvious path: integrate query syntax directly into a host language so queries get type checking and editor support ([Microsoft LINQ](https://learn.microsoft.com/en-us/dotnet/csharp/linq/)). Marrow is more radical because the host language and saved-data language are the same thing. That is a competitive advantage only if Marrow avoids a second, shadow semantic model in tooling and tests.

## 4. Alternatives Considered

Keep the current one-tree language, but improve polish. This preserves the main thesis: one syntax for local and saved resources, `^` as the saved marker, catalog-backed identities, presence proofs, source-native evolution, explicit transactions, and source-visible access cost. This is the best path if v0.1 is scoped as a small durable-app language rather than a general database/query platform.

Adopt a conventional optional type. A Rust/Swift/Kotlin-style `Option<T>` or `T?` would make local maybe-values more familiar and composable, and it would remove the special feel of sparse-path operators. It would also blur an important Marrow distinction: "the data at this saved path is absent" is not the same concept as "this local value is an optional wrapper." For v0.1, this alternative should be rejected. Later, if local optional values become necessary, they should be introduced as a separate type and not retrofitted over saved absence.

Add a declarative query language. SQL, Datalog, EdgeQL, or a LINQ-like embedded query would improve ad hoc queries, joins, projections, and planner flexibility. It would also create a second semantic layer immediately, competing with the language's core saved-path model. For v0.1, this is a reverse of the architecture laws and should be rejected. The long-term design can still add query-native analysis APIs or explicit derived indexes without promising hidden optimizer behavior.

Make histories and relations first-class database primitives. Datomic-style facts/history or Prisma-style relations/cascades would make common application modeling easier. It would also weaken Marrow's current identity story, where `Id(^store)` is an opaque typed identity and referential validity is program logic rather than engine-enforced foreign key semantics. For v0.1, keep identity simple and explicit. If referential integrity becomes common, add a source-level integrity construct later rather than smuggling FKs into the store.

Reverse `out`/`inout` into return values. This is the strongest alternative to reconsider. Modern Rust/F#/Swift/Kotlin idiom generally prefers returning values, tuples, enums, or results over mutable output parameters. Marrow's `out`/`inout` markers are explicit, but they enlarge mutation semantics, interact with presence proof invalidation, and feel less aligned with the rest of the language's simple expression model. If no user-facing examples strongly need them, `out` should be demoted or removed before the language reference freezes. `inout` for local-place mutation can remain only if it earns its cost with obvious ergonomic wins.

Make sequences dense lists and maps normal local collections. This would be much more familiar to mainstream users. It would also hide the storage shape Marrow is deliberately teaching. The better v0.1 compromise is documentation and naming discipline: describe `sequence[T]` and `map[K,V]` as saved keyed-layer conveniences, not as general-purpose arrays and maps.

Treat `merge` and `lock` as real v0.1 features. The language docs already reject this. Keeping them parsed/formatable as normal statements is the worst middle: users see the surface, tools preserve it, and the checker says no. The alternatives are either implement them with full semantics or remove them from normal syntax. For v0.1, remove them from normal parser/formatter output and keep only reservation diagnostics.

## 5. Verdict: Refine

Verdict: refine.

The core language architecture should stay. Marrow's user-facing thesis is cohesive and unusually teachable once stated plainly: write a small typed program; saved data is the same typed tree with `^`; access paths, transactions, identities, and evolution are source concepts; the engine is not a second schema/query authority. This is not a weak foundation. It is a strong foundation for v0.1 and may be optimal long-term for the intended niche of local durable programs with compiler-owned data semantics.

The current surface is not yet polished enough to freeze. Some features are merely acceptable for v0.1, not obviously optimal long-term: sparse-by-default saved fields, 1-based holey sequences, saved-only map sugar, manual history layers, no query optimizer, narrow stdlib, and debug/admin data tools. These are coherent if the docs keep the cost and storage-shape tradeoffs explicit.

There are a few weak spots that should be reversed or cleaned before more language is built on top:

- Normal parser/formatter support for `merge` and `lock` should not survive as a prototype path.
- ADRs and docs must stop teaching removed constructs such as `edit`, assertion syntax, `Book::Id`, defaulted fields, and unimplemented absence-narrowing forms.
- `out` parameters should be explicitly re-justified or removed. They are the least idiomatic part of the language surface.
- Runtime identity values should eventually carry enough store identity across dynamic/host boundaries that foreign same-shaped identities cannot survive as mere key vectors once broader `unknown` or host IO exists.

## 6. Long-Term Risks

Duplicate semantics. The accepted architecture says source, catalog, checker facts, runtime, and store each own one layer. That line is mostly respected now, but any future query API, data tool, or LSP shortcut could re-create a second resolver/classifier. The data CLI currently resolves source-shaped saved paths through checked facts, which is acceptable for admin tooling, but it must never become a production protocol contract.

Dead prototype paths. `merge` and `lock` are the visible example. They are reserved in docs, rejected by the checker, but still parsed and formatted as if they were ordinary statements. That makes the language feel larger, less honest, and more prototype-shaped than it is.

Weak Rust shape. The major runtime/store paths look better split than earlier roadmap worries, but Lane 11's broad-function and glob-prelude review still matters. A language with one semantic model cannot afford catch-all checker helpers, duplicated classifiers, or comment-heavy broad dispatch. Those are not style nits; they are where shadow semantics begin.

Hidden compatibility glue. The ADR set still carries old surface forms. Even when implementation rejects them, stale ADR prose invites future agents to revive them as "compatibility." The biggest examples are `edit`, `Book::Id`, `assert exists`, `const else`, `if let`, and default field syntax.

Unidiomatic absence teaching. The absence model is sound, but it does not match optional-value idioms in Rust, Swift, Kotlin, TypeScript, or F#. Users will guess `T?`, null, or `Option<T>`. The docs must consistently say "saved path presence" and avoid borrowing optional-value phrasing.

Sequence/map surprise. A 1-based sparse sequence with holes is a durable tree layer, not a list. A saved-only map sugar is a keyed layer, not a local map type. These concepts are coherent but easy to misread because the names are familiar.

Manual relational integrity. `Id(^store)` is a strong nominal identity type, but there is no existence guarantee, join, FK, cascade, or referential action. That is a good v0.1 simplification, but examples must show explicit checks and cleanup patterns or users will build dangling references accidentally.

Manual history modeling. History keyed child layers are transparent and controllable, but they can feel like a hand-rolled temporal database. If auditability becomes a primary value proposition, Marrow will need a source-level history/transaction-history story rather than just examples.

Destructive exact replacement. Whole-resource assignment exactly replaces data and clears omitted fields, groups, and keyed layers. That is semantically clean and cheap to explain to the compiler, but a user-facing footgun. This risk grows with richer resources and sparse fields.

Dynamic identity boundary. The docs admit that when an identity crosses through `unknown`, the runtime value shape does not carry the store root; static typing prevents same-key, different-store comparisons in typed code. That is acceptable before broad host/JSON interop, but it is a weak long-term boundary if `unknown` becomes a common interchange value.

No optimizer as a promise. The no-hidden-optimizer law is refreshingly honest. It also means Marrow must be careful not to market itself like SQL, EdgeQL, Datalog, or Prisma for ad hoc relational workloads. Source-visible cost is a language identity, not a temporary absence of engineering.

Stdlib minimalism. The narrow standard library is correct for v0.1, but practical apps will immediately ask for parsing, regex, JSON mapping, random, HTTP, and clocks beyond UTC. Host capabilities should remain explicit rather than becoming a broad standard-library escape hatch.

Roadmap trust. Lane docs are part of the architecture surface for agents. Stale Lane 10 status and stale execution-plan queues are now a real coordination risk, not just documentation lag.

## 7. Concrete Follow-Up Recommendations Ordered By Foundation Risk

1. Remove normal `merge` and `lock` syntax paths before language freeze. Keep the words reserved, but stop parsing/formatting them as ordinary statements unless there is a deliberate v0.1 implementation. This should include parser, formatter, syntax tests, checker prototype diagnostics, and docs examples.

2. Run a documentation and ADR consistency pass that deletes stale language forms rather than explaining them away. The pass should cover `edit`, `assert exists`, `Book::Id`, defaulted field syntax, `if let`, `const else`, and any wording that implies a migration DSL or compatibility bridge. Include the current unstaged cost-model ADR edits in that cleanup.

3. Re-evaluate `out` and `inout` as a language-design decision, not as inherited syntax. Preserve `inout` only if local mutation ergonomics clearly need it. Strongly consider removing `out` in favor of returned records/tuples/results before v0.1 docs make it idiomatic.

4. Add a short "What Marrow Is Not" section to the language README or roadmap: not SQL, not Datalog, not an ORM, not an optimizer, not a temporal fact database, not a JSON document store. This will make the explicit-access-path model feel principled instead of missing.

5. Add one canonical medium-sized sample that shows the whole intended idiom: resources, sparse and required fields, identity references, enum match, transaction, explicit index lookup, sequence holes, manual integrity checks, and backup/evolve boundaries. The current sample is useful, but it should carry more of the teaching load before more features accrete.

6. Rename or reframe collection docs so `sequence[T]` and `map[K,V]` are introduced as saved tree-layer conveniences. Avoid examples that make them look like dense arrays or general local maps.

7. Add diagnostics or tooling warnings for destructive whole-resource assignment when the target has keyed children, sparse groups, or indexes. The semantics should remain exact replacement, but user-facing tooling should make the blast radius visible.

8. Before broad `unknown`, host IO, JSON, or data import features, make runtime identity values store-aware or add an explicit dynamic identity check at typed reentry. Do not rely only on static typing once values can leave and reenter typed Marrow code.

9. Keep data CLI and serve protocol explicitly debug/admin. Do not add stable raw path, raw cell, or byte-level production API promises. Production tools should speak checked facts, typed diagnostics, or typed backup/restore artifacts.

10. Decide whether referential integrity stays manual through v0.1. If yes, document explicit patterns for dangling identity detection, cleanup, and transaction-time validation. If no, design source-level integrity constraints rather than adding hidden store enforcement.

11. Decide whether audit/history is a v0.1 non-goal. If yes, make manual history examples modest and do not imply Datomic-like database history. If no, design transaction history as a first-class source concept before users build incompatible manual conventions.

12. Update roadmap Lane 10 and the execution plan so agent-facing status matches git reality. Lane 11 should remain the cleanup/release-hardening gate, but it should not be carrying stale Lane 10 blockers that the integrated code and docs already addressed.

13. Keep the no-query-language decision, but revisit query-native analysis APIs for LSP/tooling separately. Editor and admin tools can need query-shaped access to checked facts without becoming a user-facing saved-data query language.

14. Preserve the narrow stdlib boundary. Add host capabilities one by one behind explicit effect typing or capability imports, not as convenience functions that silently bypass the transaction and cost model.

15. Treat examples as contract tests for idiom. Every docs example should avoid prototype syntax, raw storage terms, hidden scans, and unnecessary cleverness. If an example needs a paragraph of caveats, the feature probably needs redesign or deferral.
